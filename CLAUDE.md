# CLAUDE.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update CLAUDE.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in CLAUDE.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No API backward compat**: Project has no external users yet — don't preserve old Python APIs, function signatures, or import paths. Rename, delete, and rewrite freely; no shims or re-export modules. DB schema changes still go through Django migrations as normal — existing installs must upgrade cleanly.
- **Migrations at the end**: During a multi-file change, let the models settle first — generate migrations in one pass at the end of the sweep, then run the suite. Don't hand-write a migration mid-change.

## Project Overview

OpenOutreach — a self-hosted, **email-first** AI sales agent that learns your ICP and runs the whole funnel with **zero platform-ToS surface** (browserless; no LinkedIn account, no scraping):

**define ICP → discover → qualify → rank → find email → agentic email from your own mailbox.**

- **Discovery** is a licensed source — BetterContact **Lead Finder** (ICP search returns firmographic profiles, no emails, billed nothing).
- **Qualification** is the crown jewel — per-campaign **GPR + BALD active learning** over 384-dim embeddings, decided by an LLM.
- **Enrichment** is the one paid step — BetterContact resolves a work email for the top-ranked leads only (one credit per verified hit), fronted by a free cross-operator cache (the hub).
- **Outreach** is agentic email from mailboxes the user owns (SMTP send + IMAP reply-reading), driven by the same LLM follow-up agent. Every send carries the sending box's `Mailbox.signature` (appended in `emails/sender.py`; the agent itself never signs).

Django + Django Admin own the CRM/ORM (models are this project's); pydantic-ai drives the agents.

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **Entry**: `manage.py` — stock Django management; bare `python manage.py` (no subcommand) defaults to `rundaemon`. The `rundaemon` command (`core/management/commands/rundaemon.py`) runs: migrate (+ `setup_crm`) → onboard if incomplete → validate → `run_daemon`. **Onboarding** (`core/onboarding.py`) is interactive and email-first — product/objective → LLM (live-verified) → **mailbox** (paste app passwords → SMTP auth-check) → **signature** (per-box sign-off, asked once per never-asked box; keyed on NULL so a declined `""` sticks) → **BetterContact key** → **your email** → country → newsletter/legal. Both the mailbox and the BetterContact key are **mandatory** (BetterContact powers *both* discovery and enrichment). The `account` step asks the operator's **own email** (required) — the human's inbox, stored as `User.email` and used as the BCC-copy + newsletter target + contacts key; it is deliberately **distinct** from the mailbox `from_address` (the sending identity). The operator `User` is created only after a mailbox exists. Built as an ordered list of idempotent `Step`s (`is_done`/`run`, persist-as-you-go) that resume where they left off; interactive prompt primitives vendored in `onboarding_wizard.py` (no external `openoutreach` dep).
- **State machine**: `crm/models/deal.py:DealState` (OpenOutreach-owned `TextChoices`) — `QUALIFIED → READY_TO_FIND_EMAIL → FINDING_EMAIL → READY_TO_EMAIL → EMAILED → COMPLETED / FAILED`. The GP confidence gate promotes `QUALIFIED → READY_TO_FIND_EMAIL` (rationing the paid lookup). The paid lookup is a **two-leg async handshake** (mirroring the retired connect→check_pending): `find_email` (submit) resolves free-hub-first (hit → `READY_TO_EMAIL` directly) else fires a provider job and parks the deal at `FINDING_EMAIL`; `collect_email` (poll) then checks the job's `request_id` — hit → `READY_TO_EMAIL`, miss → `FAILED reason="no email"` with **blank outcome** so the ML labeler skips it, couldn't-run/timeout → back to `READY_TO_FIND_EMAIL`. The job handle + poll backoff live in the **collect task's payload**, never on the deal. The opener sends and parks at `EMAILED`, where the agentic follow-up loop reads replies and decides send/wait/complete until terminal. `Outcome` (converted/not_interested/wrong_fit/…) on `Deal.outcome`; `Lead.disqualified=True` = permanent account-level exclusion; LLM rejections = `FAILED` + `wrong_fit` (campaign-scoped). *(The LinkedIn connect leg — browser, Voyager, connect/check_pending, `linkedin_cli` — was removed; the `legacy` app is a model-less migration-history anchor.)*
- **Task queue**: `Task` model (persistent). Four types — `find_email`, `collect_email`, `follow_up`, `email` — handled in `openoutreach/emails/tasks/` (`handle_find_email` / `handle_collect_email` / `handle_follow_up` / `handle_email`), signature `handle_*(task, session, qualifiers)`. Two shapes of row: **lazy drains** (`find_email`/`email`/`follow_up`) carry `payload = {"campaign_id": <id>}` only and resolve their target at run time; the **bound poll** (`collect_email`) carries the in-flight lookup's `request_id`, `provider`, `submitted_at`, and backoff `attempt` in its payload. Slot creation is centralized in `core/scheduler.py` — nothing else inserts `Task` rows. There is **no spend cap or Poisson pacing**: paid spend rides on send capacity. `flush_find_email_queue` mints a submit slot only when there's mailbox send-headroom for the result *today* (`remaining_today()` minus deals already in the send pipeline, `READY_TO_EMAIL + FINDING_EMAIL`); the GP gate rations *which* leads qualify, the send cap bounds *how many* lookups ride the pipeline. `flush_email_queue` / `flush_follow_up_queue` eager-drain ready openers / due follow-ups, capped by the same per-box headroom. `collect_email` polls are **self-chaining** (each still-running poll mints its successor with doubled backoff), so one live poll exists per lookup. `Task.pending()` claims in opportunity-cost order — `follow_up` > `collect_email` > `email` > `find_email` — while `seconds_to_next()` sleeps by earliest `scheduled_at` alone (so a soon low-priority task never oversleeps behind a far-future high-priority one). `reconcile(session)` recovers crash-stale RUNNING tasks and tops up the drains — on startup and every idle cycle.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification, over 384-dim FastEmbed vectors cached on `Lead.embedding`. Per-campaign GP models in `Campaign.model_blob`.
- **Discovery + enrichment**: `openoutreach/discovery.py` (Lead Finder `search`/`embed_row`, free) uses the blocking `submit_and_poll` transport; `emails/bettercontact.py` (paid finder) splits it into `submit(query) → request_id` + `poll_once(request_id) → PollOutcome` so the daemon never blocks on a poll (the `collect_email` leg owns the backoff). `core/pipeline/`: `icp.py` (`generate_seed`: the walk's cold start — one LLM pass → the seed conjunction, taking **one value per family** so the query is a single precise conjunction, the most precise the model can name; also folds `country_code` onto the campaign and persists that conjunction as `Campaign.clauses`, **the clause pool** — with one value per family the pool *is* the seed, so the descent downstream only broadens it (drops a clause), never ORs alternatives in. The scalar `ICPSpec` fields make more than one per family unrepresentable), `frontier.py` (**discovery as a lazy best-first walk** over `DiscoveryQuery` nodes — each a **set of `Clause` rows + an `offset`**, at most one clause per family, all ANDed (`discovery.filters_for` enforces it and raises otherwise). No `params` blob, no include-lists: an OR packs several ~10k-row windows into one where separate queries get one each, for free. Only *fetched* nodes are persisted and the next query is computed lazily. A node's value is **counted, not predicted**: `node_stats` is one `GROUP BY` giving `(examined, qualified)` over its first-touch leads' deals, computed per move and **never stored** — there is no `score` column and **discovery reads no GP signal at all**. `next_query` then needs no gate, because two moves fall out of one count: **deepen** (some node has `qualified > 0` *and its own next page is still unfetched* → page that node one deeper) / **visit** (nothing qualified yet → the next unvisited conjunction, **composed** from the clause pool by `descend.py`, and only *invented* by `mutate.py` when the pool spans nothing new). Deepen is **per node, never per line**: the count is read on the exact `(clause set, offset)` node that earned it and a node earns *only* its own successor page, so a shallow page that once qualified can't keep electing its line deeper (the page it would open is already fetched) — the fix for a real non-terminating walk, where deepen used to elect the best node *anywhere* in a line and page from the line's `Max(offset)`, so an offset-0 page with 5 qualified kept that 5 forever, kept winning, and drove the offset up 100/200/300… on a `headcount 1–?` singleton that matches millions and never empties, mining the single broadest query in the lattice forever. It only ever asks whether `qualified` is positive, never ranking barren nodes against each other. The old **bootstrap**/**wall** regimes are gone (the seed is simply the head of level N), but the `examined == 0` is-*unknown*-not-barren distinction is **load-bearing** under per-node deepen: a just-fetched frontier page hasn't been ruled on, so it doesn't vote and the walk visits until its labels land — the deepen is deferred, not lost. An empty page marks that clause-set line `exhausted` — emptiness is the only thing that retires a line; a barren yield never prunes. Both signals it used to borrow were wrong for it: `min_gp_confidence` (0.9) is a **spend** bar calibrated on in-sample leads (0.755–0.829) that no out-of-sample candidate (0.121–0.327) can clear, so every node scored 0 and `deepen` never fired; and the qualifier's `n_neg > n_pos` is a **balance-driven acquisition strategy**, not a competence gate, so at a realistic base rate the walk never left bootstrap), `descend.py` (**the lattice visit as a lookup, not an LLM call** — the LLM supplies clauses, the descent composes conjunctions. **No probes anywhere**: a query execution already answers the only question a `limit=1` probe asked and pays for the rows besides, so `descend` makes **no provider call at all** — it returns the next unvisited, unpruned conjunction and `discover.py` learns by fetching it. Order: **level N → N-1 → … → 2 → 1**, then LLM refill (deepest-first, chosen 2026-07-17). Level N opens it because a long conjunction matches fewer people and reaches past the provider's famous-company head (`{lead_seniority: founder}` → Meta, Meta, Meta) straight into the niche; the seed is the head of level N, so the walk opens on the ICP's strongest guess. Shallow levels come later (they widen back toward that head) and singletons last of all. The cost is pruning: the subset test only bites when the recorded sets are **shorter** than the candidates, and deepest-first records the singletons *last*, so a globally-dead clause like `lead_location: Europe` isn't convicted until its singleton is reached and sits in every deep conjunction holding it (each fetched, each empty) until then — the accepted cost of reaching the niche first; the subset table then pays mostly across refills. Live 19-clause pool measures 1175 nodes `{1: 19, 5: 180, 4: 456, 3: 389, 2: 131}`, but `deepen` pre-empts the visit as soon as anything qualifies. Empty conjunctions (at **offset 0** only — deeper is a vein running out) go to `EmptyClauseSet`, whose **subset test** (`a candidate is dead iff some recorded set ⊆ it`) prunes their supersets free. **Only emptiness prunes; yield never does** — a barren node means *those people aren't our ICP* and retires nothing, and an empty conjunction convicts the whole set, never a clause inside it (`lead_department: Sales` returns rows alone yet sat in six 0-row conjunctions — a clause is only retired by its own singleton coming back empty, which is sound). `[]` means the pool spans nothing new, the one condition that licenses an LLM call), `mutate.py` (the swappable generator seam, defaulting to `descend_or_refill`: **next unvisited conjunction → (only when none remain) → LLM refill**. The refill's typed `_Filters` output takes **one value per family** — a list would be an OR, and an OR is dominated — and carrying **only families verified to steer** — each probed live against an unfiltered baseline *with an absurd-value control*, 2026-07-16. An unknown *key* is **silently dropped**: you get the unfiltered page back, *with rows*, so it reads as success — worse than an error. `lead_industry`, `company_technology` and `lead_skills` were exactly that (`lead_industry` rode every seed) and are gone; `extra="forbid"` keeps them unrepresentable, since pydantic otherwise *ignores* unknown keys and would reproduce the same silent drop in our own code. An unknown *value* is benign — an empty page, one move. Only `lead_seniority` is a closed vocabulary (`discovery.Seniority`, a `Literal`); `lead_department`/`lead_function` are free text despite a published "enum" that matches nothing (`Sales`, `Marketing` match; `Growth` doesn't), and `lead_location` takes real country names — `Europe` matches zero leads), `discover.py` (fetch the selected query's page into first-touch `Lead`s; takes no qualifier), `qualify.py` → `ready_pool.py` (GP gate; `min_gp_confidence` is the paid-lookup spend gate **and nothing else** — the frontier used to borrow it as a node score and that is exactly what broke the walk) → `pools.py` (the composable generator chain, running in one of **two states** decided per label by whether any unlabelled lead reaches `min_gp_confidence`: **consume** (some do → qualify them; they can clear the promote gate and reach email — having any is what it *means* to be out of the cold start) or **cold start** (none do → discover a page and spend exactly one label on the pool, chosen with no threshold, since nothing meeting it is what put us here). The threshold is deliberately the *same constant* the promote gate uses, and that is the whole argument for it: this is not an "is this pool promising?" judgment but the fact that a lead below it **parks** — it would be qualified and then blocked by `promote_to_ready`, so the call buys a label and nothing else. In cold start the label is the point. Measured 2026-07-17: the pool tops out at 0.327, so this runs in cold start until many more labels exist — expected at 4 positives, and a lead the LLM accepts meanwhile parks at QUALIFIED unemailed, having done its job. Two bars that *were* judgments failed and are not to return: `max(0, 0.20 - 1/sqrt(n_obs))` is capped at 0.20 while the pool's best lead scores 0.327 (never fired), and `positive_score_floor(25)` compared **in-sample** GP scores (0.755–0.829, its own training points) against **out-of-sample** ones (0.121–0.327) and so fired *always*, running discovery 17× ahead of qualification. A fitted GP reproduces its training points and regresses the unseen toward the prior, so no bar drawn from one population applies to the other — which is why this gate compares against the *next gate's* constant instead of inventing one. Discovery steering belongs to the frontier, on ground-truth per-node counts — see the discovery-query-graph-search card).
- **Contacts store (hub)**: `contacts/service.py` — an optional free `profile_url → email` cache at `hub.openoutreach.app`, tried *before* the paid finder and given back to on a fresh paid hit. Both calls are best-effort (outage/no-token → no-op) and gated on `has_mailbox()`. The give-back is **non-EEA only** and derived from the operator's onboarding country (`not is_eea_located`) — never a stored toggle; the server re-gates authoritatively. A per-operator token is earned on first contribution and stored in `SiteConfig.contacts_api_token` (never the repo).
- **Config**: `SiteConfig` DB singleton — `ai_model` (a pydantic-ai `provider:model` id, e.g. `anthropic:claude-sonnet-4-5-20250929`; bare `gpt-*`/`o1`/`o3`/`claude-*`/`gemini-*` auto-prefixed), `llm_api_key`, `llm_api_base` (only for `openai_compatible:*`), `bettercontact_api_key` (blank disables discovery + enrichment), `contacts_api_token`/`contacts_api_url`, `country_code` (the only persisted operator setting — drives timezone + email jurisdiction). `core/conf.py`: `CAMPAIGN_CONFIG` (ML + human-rhythm defaults, incl. `min_gp_confidence` for the paid-step rank gate — which is **only** a spend gate; there is no discovery-interleave threshold), `COLLECT_BACKOFF_BASE_S`/`COLLECT_BACKOFF_MAX_S`/`COLLECT_DEADLINE_S` (the `collect_email` poll backoff — paid spend is gated by send-headroom, not a cap), `DEFAULT_EMAIL_DAILY_LIMIT`, active-hours knobs (`ENABLE_ACTIVE_HOURS`/`ACTIVE_*`; `ACTIVE_TIMEZONE=None` → resolved from the operator's country).
- **Django apps** (all nested under `openoutreach/`, dotted `AppConfig.name`, short labels): `core` (engine — daemon, task queue + scheduler, Campaign/SiteConfig/Task, llm, onboarding, ML, discovery/qualify pipeline, the two agents), `crm` (Lead + Deal), `chat` (ChatMessage — the per-Deal conversation), `emails` (discovery/enrichment client, Mailbox + import + SMTP/IMAP, sender, the three task handlers), `legacy` (model-less migration-history anchor). `contacts` is a service-only module (no models, not an installed app). One engine, one channel.
- **Data dir**: `data/` holds `db.sqlite3`. Docker mounts a volume at `/app/data`.
- **Docker**: `python:3.12-slim` multi-stage build with `uv`; no browser, no VNC. `compose/openoutreach/Dockerfile`. `BUILD_ENV` arg selects requirements.
- **CI/CD**: `.github/workflows/tests.yml` (pytest), `deploy.yml` (**every push to `main`**, plus `v*` tags → `make docker-test` → build + push `ghcr.io/eracle/openoutreach` tagged `latest` + `sha-<pushed tip>` → `repository-dispatch: image-updated` to the hub repo). **There is no release gate: merging to `main` republishes `:latest`**, so code *and schema migrations* reach anyone pulling `latest` on merge, not on a tag. No `v*` tag has ever been cut, so no semver tag exists to pin or roll back to.
