# CLAUDE.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update CLAUDE.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in CLAUDE.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No API backward compat**: Project has no external users yet ÔÇö don't preserve old Python APIs, function signatures, or import paths. Rename, delete, and rewrite freely; no shims or re-export modules. DB schema changes still go through Django migrations as normal ÔÇö existing installs must upgrade cleanly.
- **Migrations at the end**: During a multi-file change, let the models settle first ÔÇö generate migrations in one pass at the end of the sweep, then run the suite. Don't hand-write a migration mid-change.

## Project Overview

OpenOutreach ÔÇö a self-hosted, **email-first** AI sales agent that learns your ICP and runs the whole funnel with **zero platform-ToS surface** (browserless; no LinkedIn account, no scraping):

**define ICP Ôćĺ discover Ôćĺ qualify Ôćĺ rank Ôćĺ find email Ôćĺ agentic email from your own mailbox.**

- **Discovery** is a licensed source ÔÇö BetterContact **Lead Finder** (ICP search returns firmographic profiles, no emails, billed nothing).
- **Qualification** is the crown jewel ÔÇö per-campaign **GPR + BALD active learning** over 384-dim embeddings, decided by an LLM.
- **Enrichment** is the one paid step ÔÇö BetterContact resolves a work email for the top-ranked leads only (one credit per verified hit), fronted by a free cross-operator cache (the hub).
- **Outreach** is agentic email from mailboxes the user owns (SMTP send + IMAP reply-reading), driven by the same LLM follow-up agent. Every send carries the sending box's `Mailbox.signature` (appended in `emails/sender.py`; the agent itself never signs).

Django + Django Admin own the CRM/ORM (models are this project's); pydantic-ai drives the agents.

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **Entry**: `manage.py` ÔÇö stock Django management; bare `python manage.py` (no subcommand) defaults to `rundaemon`. The `rundaemon` command (`core/management/commands/rundaemon.py`) runs: migrate (+ `setup_crm`) Ôćĺ onboard if incomplete Ôćĺ validate Ôćĺ `run_daemon`. **Onboarding** (`core/onboarding.py`) is interactive and email-first ÔÇö product/objective Ôćĺ LLM (live-verified) Ôćĺ **mailbox** (paste app passwords Ôćĺ SMTP auth-check) Ôćĺ **signature** (per-box sign-off, asked once per never-asked box; keyed on NULL so a declined `""` sticks) Ôćĺ **BetterContact key** Ôćĺ **your email** Ôćĺ country Ôćĺ newsletter/legal. Both the mailbox and the BetterContact key are **mandatory** (BetterContact powers *both* discovery and enrichment). The `account` step asks the operator's **own email** (required) ÔÇö the human's inbox, stored as `User.email` and used as the BCC-copy + newsletter target + contacts key; it is deliberately **distinct** from the mailbox `from_address` (the sending identity). The operator `User` is created only after a mailbox exists. Built as an ordered list of idempotent `Step`s (`is_done`/`run`, persist-as-you-go) that resume where they left off; interactive prompt primitives vendored in `onboarding_wizard.py` (no external `openoutreach` dep).
- **State machine**: `crm/models/deal.py:DealState` (OpenOutreach-owned `TextChoices`) ÔÇö `QUALIFIED Ôćĺ READY_TO_FIND_EMAIL Ôćĺ FINDING_EMAIL Ôćĺ READY_TO_EMAIL Ôćĺ EMAILED Ôćĺ COMPLETED / FAILED`. The GP confidence gate promotes `QUALIFIED Ôćĺ READY_TO_FIND_EMAIL` (rationing the paid lookup). The paid lookup is a **two-leg async handshake** (mirroring the retired connectÔćĺcheck_pending): `find_email` (submit) resolves free-hub-first (hit Ôćĺ `READY_TO_EMAIL` directly) else fires a provider job and parks the deal at `FINDING_EMAIL`; `collect_email` (poll) then checks the job's `request_id` ÔÇö hit Ôćĺ `READY_TO_EMAIL`, miss Ôćĺ `FAILED reason="no email"` with **blank outcome** so the ML labeler skips it, couldn't-run/timeout Ôćĺ back to `READY_TO_FIND_EMAIL`. The job handle + poll backoff live in the **collect task's payload**, never on the deal. The opener sends and parks at `EMAILED`, where the agentic follow-up loop reads replies and decides send/wait/complete until terminal. `Outcome` (converted/not_interested/wrong_fit/ÔÇŽ) on `Deal.outcome`; `Lead.disqualified=True` = permanent account-level exclusion; LLM rejections = `FAILED` + `wrong_fit` (campaign-scoped). *(The LinkedIn connect leg ÔÇö browser, Voyager, connect/check_pending, `linkedin_cli` ÔÇö was removed; the `legacy` app is a model-less migration-history anchor.)*
- **Task queue**: `Task` model (persistent). Four types ÔÇö `find_email`, `collect_email`, `follow_up`, `email` ÔÇö handled in `openoutreach/emails/tasks/` (`handle_find_email` / `handle_collect_email` / `handle_follow_up` / `handle_email`), signature `handle_*(task, session, qualifiers)`. Two shapes of row: **lazy drains** (`find_email`/`email`/`follow_up`) carry `payload = {"campaign_id": <id>}` only and resolve their target at run time; the **bound poll** (`collect_email`) carries the in-flight lookup's `request_id`, `provider`, `submitted_at`, and backoff `attempt` in its payload. Slot creation is centralized in `core/scheduler.py` ÔÇö nothing else inserts `Task` rows. There is **no spend cap or Poisson pacing**: paid spend rides on send capacity. `flush_find_email_queue` mints a submit slot only when there's mailbox send-headroom for the result *today* (`remaining_today()` minus deals already in the send pipeline, `READY_TO_EMAIL + FINDING_EMAIL`); the GP gate rations *which* leads qualify, the send cap bounds *how many* lookups ride the pipeline. `flush_email_queue` / `flush_follow_up_queue` eager-drain ready openers / due follow-ups, capped by the same per-box headroom. `collect_email` polls are **self-chaining** (each still-running poll mints its successor with doubled backoff), so one live poll exists per lookup. `Task.pending()` claims in opportunity-cost order ÔÇö `follow_up` > `collect_email` > `email` > `find_email` ÔÇö while `seconds_to_next()` sleeps by earliest `scheduled_at` alone (so a soon low-priority task never oversleeps behind a far-future high-priority one). `reconcile(session)` recovers crash-stale RUNNING tasks and tops up the drains ÔÇö on startup and every idle cycle.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification, over 384-dim FastEmbed vectors cached on `Lead.embedding`. Per-campaign GP models in `Campaign.model_blob`.
- **Discovery + enrichment**: `openoutreach/discovery.py` (Lead Finder `search`/`embed_row`, free) uses the blocking `submit_and_poll` transport; `emails/bettercontact.py` (paid finder) splits it into `submit(query) Ôćĺ request_id` + `poll_once(request_id) Ôćĺ PollOutcome` so the daemon never blocks on a poll (the `collect_email` leg owns the backoff). `core/pipeline/`: `icp.py` (`generate_seed`: the walk's cold start ÔÇö one LLM pass Ôćĺ the seed conjunction, taking **one value per family** so the query is a single precise conjunction, the most precise the model can name; also folds `country_code` onto the campaign and persists that conjunction as `Campaign.clauses`, **the clause pool** ÔÇö with one value per family the pool *is* the seed, so the descent downstream only broadens it (drops a clause), never ORs alternatives in. The scalar `ICPSpec` fields make more than one per family unrepresentable), `frontier.py` (**discovery as a lazy best-first walk** over `DiscoveryQuery` nodes ÔÇö each a **set of `Clause` rows + an `offset`**, at most one clause per family, all ANDed (`discovery.filters_for` enforces it and raises otherwise). No `params` blob, no include-lists: an OR packs several ~10k-row windows into one where separate queries get one each, for free. Only *fetched* nodes are persisted and the next query is computed lazily. A query's value is **counted, not predicted**: `line_stats` is one `GROUP BY` giving `(examined, qualified)` per conjunction ÔÇö summed over *all its offsets* (`node_stats` gives the same per page, for the refill prompt) ÔÇö computed per move and **never stored** (no `score` column, and **discovery reads no GP signal at all**). Two moves fall out of that count, interleaved **1:1**: **deepen** (a conjunction has `qualified > 0` and its line is still open Ôćĺ page it one offset deeper) / **visit** (open the next conjunction from `descend.py`, `mutate.py` inventing fresh clauses only when the pool spans nothing new). Deepen is **per conjunction, summed over offsets**: a lead qualified at offset 0 keeps paging the line deeper and doesn't expire when a deep page qualifies nobody ÔÇö the reversal of an earlier **per-node** rule that stopped a non-terminating walk (paging from a line's `Max(offset)` mined a `headcount 1ÔÇô?` singleton forever, since a broad singleton never empties). Safe to drop now because `descend.py` is **deepest-only** (the only lines are deep conjunctions, which empty in bounded pages) and the 1:1 interleave caps any one line at half the moves. The 1:1 balance is stateless strict alternation: deepen leaves a page at `offset > 0`, visit a conjunction at `offset == 0`, so the newest node's offset says which came last and `next_query` takes the other ÔÇö deepen is unavailable until something qualifies, so cold-start moves are all visits, with no catch-up burst. The old **bootstrap**/**wall** regimes are gone (the seed is simply the deepest conjunction the pool spans), but the `examined == 0` is-*unknown*-not-barren distinction is **load-bearing**: a just-fetched page doesn't vote, so cold-start moves fall to `visit` until its labels land ÔÇö the deepen is deferred, not lost. An empty page marks that clause-set line `exhausted` ÔÇö emptiness is the only thing that retires a line; a barren yield never prunes. Both signals it used to borrow were wrong for it: `min_gp_confidence` (0.9) is a **spend** bar calibrated on in-sample leads (0.755ÔÇô0.829) that no out-of-sample candidate (0.121ÔÇô0.327) can clear, so every node scored 0 and `deepen` never fired; and the qualifier's `n_neg > n_pos` is a **balance-driven acquisition strategy**, not a competence gate, so at a realistic base rate the walk never left bootstrap), `descend.py` (**the lattice visit as a lookup, not an LLM call** ÔÇö the LLM supplies clauses, the descent composes conjunctions. **No probes anywhere**: a query execution already answers the only question a `limit=1` probe asked and pays for the rows besides, so `descend` makes **no provider call at all** ÔÇö it returns the deepest unfetched conjunction and `discover.py` learns by fetching it. **Deepest-only**: it returns the **maximal** conjunctions (one value from every family) and nothing shallower, because a shorter conjunction is a strictly broader window over the same corpus (a superset of the deeper query's people, drifting toward the provider's famous-company head ÔÇö `{lead_seniority: founder}` Ôćĺ Meta, Meta, Meta); volume from a working query comes from paging it deeper, not dropping clauses. A conjunction is widened for exactly one reason ÔÇö **its deeper parent matched nobody**: an empty query unlocks its drop-one children (`{a,b,c}` Ôćĺ `{a,b}`, `{a,c}`, `{b,c}`), recursively, until queries match people (anti-monotonicity forward ÔÇö an empty query's supersets are empty, its subsets may not be). So `EmptyClauseSet` (offset 0 only ÔÇö deeper is a vein running out) does double duty: its **subset test** (`a candidate is dead iff some recorded set ÔŐć it`) prunes supersets free, **and** the same emptiness unlocks that conjunction's children. **Only emptiness widens; yield never does** ÔÇö a barren node means *those people aren't our ICP*, so widening it only adds more of the wrong people, and an empty conjunction convicts the whole set, never a clause inside it (`lead_department: Sales` returns rows alone yet sat in six 0-row conjunctions). `[]` means the pool spans nothing new ÔÇö the refill then adds clauses that recombine into new maximal conjunctions, so the deepest layer grows rather than the walk drifting shallow), `mutate.py` (the swappable generator seam, defaulting to `descend_or_refill`: **next unvisited conjunction Ôćĺ (only when none remain) Ôćĺ LLM refill**. The refill's typed `_Filters` output takes **one value per family** ÔÇö a list would be an OR, and an OR is dominated ÔÇö and carrying **only families verified to steer** ÔÇö each probed live against an unfiltered baseline *with an absurd-value control*, 2026-07-16. An unknown *key* is **silently dropped**: you get the unfiltered page back, *with rows*, so it reads as success ÔÇö worse than an error. `lead_industry`, `company_technology` and `lead_skills` were exactly that (`lead_industry` rode every seed) and are gone; `extra="forbid"` keeps them unrepresentable, since pydantic otherwise *ignores* unknown keys and would reproduce the same silent drop in our own code. An unknown *value* is benign ÔÇö an empty page, one move. Only `lead_seniority` is a closed vocabulary (`discovery.Seniority`, a `Literal`); `lead_department`/`lead_function` are free text despite a published "enum" that matches nothing (`Sales`, `Marketing` match; `Growth` doesn't), and `lead_location` takes real country names ÔÇö `Europe` matches zero leads), `discover.py` (fetch the selected query's page into first-touch `Lead`s; takes no qualifier), `qualify.py` Ôćĺ `ready_pool.py` (GP gate; `min_gp_confidence` is the paid-lookup spend gate **and nothing else** ÔÇö the frontier used to borrow it as a node score and that is exactly what broke the walk) Ôćĺ `pools.py` (the composable generator chain, running in one of **two states** decided per label by whether any unlabelled lead reaches `min_gp_confidence`: **consume** (some do Ôćĺ qualify them; they can clear the promote gate and reach email ÔÇö having any is what it *means* to be out of the cold start) or **cold start** (none do Ôćĺ discover a page and spend exactly one label on the pool, chosen with no threshold, since nothing meeting it is what put us here). The threshold is deliberately the *same constant* the promote gate uses, and that is the whole argument for it: this is not an "is this pool promising?" judgment but the fact that a lead below it **parks** ÔÇö it would be qualified and then blocked by `promote_to_ready`, so the call buys a label and nothing else. In cold start the label is the point. Measured 2026-07-17: the pool tops out at 0.327, so this runs in cold start until many more labels exist ÔÇö expected at 4 positives, and a lead the LLM accepts meanwhile parks at QUALIFIED unemailed, having done its job. Two bars that *were* judgments failed and are not to return: `max(0, 0.20 - 1/sqrt(n_obs))` is capped at 0.20 while the pool's best lead scores 0.327 (never fired), and `positive_score_floor(25)` compared **in-sample** GP scores (0.755ÔÇô0.829, its own training points) against **out-of-sample** ones (0.121ÔÇô0.327) and so fired *always*, running discovery 17├Ś ahead of qualification. A fitted GP reproduces its training points and regresses the unseen toward the prior, so no bar drawn from one population applies to the other ÔÇö which is why this gate compares against the *next gate's* constant instead of inventing one. Discovery steering belongs to the frontier, on ground-truth per-conjunction counts ÔÇö see the discovery-query-graph-search card).
- **Contacts store (hub)**: `contacts/service.py` ÔÇö an optional free `profile_url Ôćĺ email` cache at `hub.openoutreach.app`, tried *before* the paid finder and given back to on a fresh paid hit. Both calls are best-effort (outage/no-token Ôćĺ no-op) and gated on `has_mailbox()`. The give-back is **non-EEA only** and derived from the operator's onboarding country (`not is_eea_located`) ÔÇö never a stored toggle; the server re-gates authoritatively. A per-operator token is earned on first contribution and stored in `SiteConfig.contacts_api_token` (never the repo).
- **Config**: `SiteConfig` DB singleton ÔÇö `ai_model` (a pydantic-ai `provider:model` id, e.g. `anthropic:claude-sonnet-4-5-20250929`; bare `gpt-*`/`o1`/`o3`/`claude-*`/`gemini-*` auto-prefixed), `llm_api_key`, `llm_api_base` (only for `openai_compatible:*`), `bettercontact_api_key` (blank disables discovery + enrichment), `contacts_api_token`/`contacts_api_url`, `country_code` (the only persisted operator setting ÔÇö drives timezone + email jurisdiction). `core/conf.py`: `CAMPAIGN_CONFIG` (ML + human-rhythm defaults, incl. `min_gp_confidence` for the paid-step rank gate ÔÇö which is **only** a spend gate; there is no discovery-interleave threshold), `COLLECT_BACKOFF_BASE_S`/`COLLECT_BACKOFF_MAX_S`/`COLLECT_DEADLINE_S` (the `collect_email` poll backoff ÔÇö paid spend is gated by send-headroom, not a cap), `DEFAULT_EMAIL_DAILY_LIMIT`, active-hours knobs (`ENABLE_ACTIVE_HOURS`/`ACTIVE_*`; `ACTIVE_TIMEZONE=None` Ôćĺ resolved from the operator's country).
- **Django apps** (all nested under `openoutreach/`, dotted `AppConfig.name`, short labels): `core` (engine ÔÇö daemon, task queue + scheduler, Campaign/SiteConfig/Task, llm, onboarding, ML, discovery/qualify pipeline, the two agents), `crm` (Lead + Deal), `chat` (ChatMessage ÔÇö the per-Deal conversation), `emails` (discovery/enrichment client, Mailbox + import + SMTP/IMAP, sender, the three task handlers), `legacy` (model-less migration-history anchor). `contacts` is a service-only module (no models, not an installed app). One engine, one channel.
- **Data dir**: `data/` holds `db.sqlite3`. Docker mounts a volume at `/app/data`.
- **Docker**: `python:3.12-slim` multi-stage build with `uv`; no browser, no VNC. `compose/openoutreach/Dockerfile`. `BUILD_ENV` arg selects requirements.
- **CI/CD**: `.github/workflows/tests.yml` (pytest), `deploy.yml` (**every push to `main`**, plus `v*` tags Ôćĺ `make docker-test` Ôćĺ build + push `ghcr.io/eracle/openoutreach` tagged `latest` + `sha-<pushed tip>` Ôćĺ `repository-dispatch: image-updated` to the hub repo). **There is no release gate: merging to `main` republishes `:latest`**, so code *and schema migrations* reach anyone pulling `latest` on merge, not on a tag. No `v*` tag has ever been cut, so no semver tag exists to pin or roll back to.
