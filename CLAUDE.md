# CLAUDE.md

## Rules

- **Python env**: Always use `.venv/bin/python` (not system `python3`).
- **Commits**: No `Co-Authored-By` lines. Single-line messages (no body).
- **Dependencies**: Managed in `requirements/*.txt` (used by local dev and Docker).
- **Docs sync**: When modifying code, update CLAUDE.md and ARCHITECTURE.md to reflect changes.
- **No memory**: Never use the auto-memory system (no MEMORY.md, no memory files). All persistent context belongs in CLAUDE.md or ARCHITECTURE.md.
- **Error handling**: App should crash on unexpected errors. `try/except` only for expected, recoverable errors. Custom exceptions in `exceptions.py`.
- **No API backward compat**: Project has no external users yet ‚Äî don't preserve old Python APIs, function signatures, or import paths. Rename, delete, and rewrite freely; no shims or re-export modules. DB schema changes still go through Django migrations as normal ‚Äî existing installs must upgrade cleanly.
- **Migrations at the end**: During a multi-file change, let the models settle first ‚Äî generate migrations in one pass at the end of the sweep, then run the suite. Don't hand-write a migration mid-change.

## Project Overview

OpenOutreach ‚Äî a self-hosted, **email-first** AI sales agent that learns your ICP and runs the whole funnel with **zero platform-ToS surface** (browserless; no LinkedIn account, no scraping):

**define ICP ‚Üí discover ‚Üí qualify ‚Üí rank ‚Üí find email ‚Üí agentic email from your own mailbox.**

- **Discovery** is a licensed source ‚Äî BetterContact **Lead Finder** (ICP search returns firmographic profiles, no emails, billed nothing).
- **Qualification** is the crown jewel ‚Äî per-campaign **GPR + BALD active learning** over 384-dim embeddings, decided by an LLM.
- **Enrichment** is the one paid step ‚Äî BetterContact resolves a work email for the top-ranked leads only (one credit per verified hit), fronted by a free cross-operator cache (the hub).
- **Outreach** is agentic email from mailboxes the user owns (SMTP send + IMAP reply-reading), driven by the same LLM follow-up agent. Every send carries the sending box's `Mailbox.signature` (appended in `emails/sender.py`; the agent itself never signs).

Django + Django Admin own the CRM/ORM (models are this project's); pydantic-ai drives the agents.

## Architecture (quick reference)

For detailed module docs, see `ARCHITECTURE.md`.

- **Entry**: `manage.py` ‚Äî stock Django management; bare `python manage.py` (no subcommand) defaults to `rundaemon`. The `rundaemon` command (`core/management/commands/rundaemon.py`) runs: migrate (+ `setup_crm`) ‚Üí onboard if incomplete ‚Üí validate ‚Üí `run_daemon`. **Onboarding** (`core/onboarding.py`) is interactive and email-first ‚Äî product/objective ‚Üí LLM (live-verified) ‚Üí **mailbox** (paste app passwords ‚Üí SMTP auth-check) ‚Üí **signature** (per-box sign-off, asked once per never-asked box; keyed on NULL so a declined `""` sticks) ‚Üí **BetterContact key** ‚Üí **your email** ‚Üí country ‚Üí newsletter/legal. Both the mailbox and the BetterContact key are **mandatory** (BetterContact powers *both* discovery and enrichment). The `account` step asks the operator's **own email** (required) ‚Äî the human's inbox, stored as `User.email` and used as the newsletter target + contacts key + **optional** BCC-copy target (gated by `conf.BCC_OPERATOR_ON_SEND`, off by default); it is deliberately **distinct** from the mailbox `from_address` (the sending identity). The operator `User` is created only after a mailbox exists. Built as an ordered list of idempotent `Step`s (`is_done`/`run`, persist-as-you-go) that resume where they left off; interactive prompt primitives vendored in `onboarding_wizard.py` (no external `openoutreach` dep).
- **State machine**: `crm/models/deal.py:DealState` (OpenOutreach-owned `TextChoices`) ‚Äî `QUALIFIED ‚Üí READY_TO_FIND_EMAIL ‚Üí FINDING_EMAIL ‚Üí READY_TO_EMAIL ‚Üí EMAILED ‚Üí COMPLETED / FAILED`. The GP confidence gate promotes `QUALIFIED ‚Üí READY_TO_FIND_EMAIL` (rationing the paid lookup). The paid lookup is a **two-leg async handshake** (mirroring the retired connect‚Üícheck_pending): `find_email` (submit) resolves free-hub-first (hit ‚Üí `READY_TO_EMAIL` directly) else fires a provider job and parks the deal at `FINDING_EMAIL`; `collect_email` (poll) then checks the job's `request_id` ‚Äî hit ‚Üí `READY_TO_EMAIL`, miss ‚Üí `FAILED reason="no email"` with **blank outcome** so the ML labeler skips it, couldn't-run/timeout ‚Üí back to `READY_TO_FIND_EMAIL`. The job handle + poll backoff live in the **collect task's payload**, never on the deal. The opener sends and parks at `EMAILED`, where the agentic follow-up loop reads replies and decides send/wait/complete until terminal. `Outcome` (converted/not_interested/wrong_fit/‚Ä¶) on `Deal.outcome`; `Lead.disqualified=True` = permanent account-level exclusion; LLM rejections = `FAILED` + `wrong_fit` (campaign-scoped). *(The LinkedIn connect leg ‚Äî browser, Voyager, connect/check_pending, `linkedin_cli` ‚Äî was removed; the `legacy` app is a model-less migration-history anchor.)*
- **Task queue**: `Task` model (persistent). Four types ‚Äî `find_email`, `collect_email`, `follow_up`, `email` ‚Äî handled in `openoutreach/emails/tasks/` (`handle_find_email` / `handle_collect_email` / `handle_follow_up` / `handle_email`), signature `handle_*(task, session, qualifiers)`. Two shapes of row: **lazy drains** (`find_email`/`email`/`follow_up`) carry `payload = {"campaign_id": <id>}` only and resolve their target at run time; the **bound poll** (`collect_email`) carries the in-flight lookup's `request_id`, `provider`, `submitted_at`, and backoff `attempt` in its payload. Slot creation is centralized in `core/scheduler.py` ‚Äî nothing else inserts `Task` rows. There is **no spend cap or Poisson pacing**: paid spend rides on send capacity. `flush_find_email_queue` mints a submit slot only when there's mailbox send-headroom for the result *today* (`remaining_today()` minus deals already in the send pipeline, `READY_TO_EMAIL + FINDING_EMAIL`); the GP gate rations *which* leads qualify, the send cap bounds *how many* lookups ride the pipeline. `flush_email_queue` / `flush_follow_up_queue` eager-drain ready openers / due follow-ups, capped by the same per-box headroom. `collect_email` polls are **self-chaining** (each still-running poll mints its successor with doubled backoff), so one live poll exists per lookup. `Task.pending()` claims in opportunity-cost order ‚Äî `follow_up` > `collect_email` > `email` > `find_email` ‚Äî while `seconds_to_next()` sleeps by earliest `scheduled_at` alone (so a soon low-priority task never oversleeps behind a far-future high-priority one). `reconcile(session)` recovers crash-stale RUNNING tasks and tops up the drains ‚Äî on startup and every idle cycle.
- **ML pipeline**: GPR (sklearn) + BALD active learning + LLM qualification, over 384-dim FastEmbed vectors cached on `Lead.embedding`. Per-campaign GP models in `Campaign.model_blob`.
- **Discovery + enrichment**: `openoutreach/discovery.py` (Lead Finder `search`, free) uses the blocking `submit_and_poll` transport; `emails/bettercontact.py` (paid finder) splits it into `submit(query) ‚Üí request_id` + `poll_once(request_id) ‚Üí PollOutcome` so the daemon never blocks on a poll (the `collect_email` leg owns the backoff). **Discovery is one GP-scored walk over *maximal* queries** ‚Äî from first principles: clauses are the axes, and the primary queries are maximals (one value per family, the Cartesian product of the campaign's clause pool). Growth comes from *more clause values* (minting), never from dropping a clause at composition time; the one loosening is **lazy backoff** ‚Äî a conjunction that matches nobody enqueues its one-clause-looser generalizations, harvested like any query (no diagnostic-only fetch). Every new clause value is **pre-screened** first (fetched **truly alone** ‚Äî no headcount band, no sibling clause; the check is only whether the keyword means anything to Lead Finder ‚Äî its full page harvested, 0-support values dropped and recorded empty **as a size-1 set of the value alone**, which is sound to record *globally* precisely because the probe convicts the value itself, not the value-within-this-band; a pruned value generates no backoff child and can't be resurrected into an un-rankable candidate), so no maximal is ever composed from a dead axis. `core/pipeline/`: `icp.py` (`generate_seed`: the cold start ‚Äî one LLM pass ‚Üí the seed conjunction, **one value per family** (scalar `ICPSpec`), folded onto `Campaign.clauses`; the seed is the single most-precise starting query and the whole starting pool ‚Äî one maximal), `select.py` (**the selector, and the whole steering**: `next_query(campaign, qualifier)` builds the candidate frontier ‚Äî the pool's maximals **plus the one-clause-looser generalizations of every recorded empty** (`_generalizations`, the backoff), deduped, minus exhausted lines and empty-pruned sets ‚Äî each either a **fresh** set (offset 0) or a **deepen** of a fetched, non-exhausted one (its next page). Candidates are scored the *same* way: embed the keywords (`discovery.embed_query`) and read the GP's balance-driven `qualifier.acquisition_scores` (predicted P in exploit mode, BALD info-gain in explore). Argmax fetches. Exact-embedding every maximal is too costly at scale, so `_prefilter` ranks the **whole** pool by a cheap composed score (embed only the pool's distinct clause phrases, pool them per query) and exact-embeds only the top-K on the live axis (`qualifier.acquisition_mode`, `PREFILTER_K`) ‚Äî a small K for exploit (mean-pooling tracks P, complete by K‚âà128), a larger K for explore (BALD's variance doesn't decompose over clauses). Both proxies are normalized by clause count so mixed-depth candidates (backoff admits sub-maximals) compare depth-neutrally, not by term count. There is **no deepen/visit alternation and no counted-deal metric**: the GP that ranks which lead to label ranks which query to fetch, because a discovered lead carries its retrieving query's keywords in its embedding (see keyword injection). Deepen-vs-explore is argmax over one score, and a vein self-bounds ‚Äî a maximal empties within the 10k window and is marked `exhausted`. Cold start (GP unfitted ‚Üí `acquisition_scores` None) falls back to seed-first, fresh-before-deep. `None` means the pool spans nothing fetchable ‚Äî the saturation signal. Also carries the node primitives `persist_fetched`/`mark_exhausted`/`record_empty`/`clause_key`), `mint.py` (`mint_clauses`: grow the axes from the **qualified** leads ‚Äî the LLM reads their real titles/seniorities/locations and proposes new clause *values* adjacent to them (a new value widens the product; a new family deepens every maximal). Triggered two ways by `discover`, **neither a GP-confidence gate** (which would deadlock cold start): **throughput** ‚Äî every `CAMPAIGN_CONFIG["mint_every_n_qualified"]` new qualified leads (a count, tracked by `Campaign.discovery_minted_at_qualified`); **saturation** ‚Äî the selector returns `None`, so widen once and stop only if nothing new is added. Best-effort: an LLM outage adds nothing and the walk carries on), `discover.py` (`discover(session, qualifier)`: seed on cold start ‚Üí **pre-screen** new values (`_prescreen`: fetch each alone, drop 0-support, harvest the rest) ‚Üí throughput mint ‚Üí fetch the GP's top candidate into first-touch `Lead`s (`_harvest`) ‚Üí record the two shapes of empty (offset 0 blacklists the conjunction *and* backs off to its generalizations, deeper exhausts the vein) ‚Üí saturation mint. Takes the qualifier because the GP is the selector), `qualify.py` ‚Üí `ready_pool.py` (GP gate; `min_gp_confidence` is the paid-lookup spend gate **and nothing else**) ‚Üí `pools.py` (`find_candidate` loops three moves to surface one ready lead ‚Äî hand off a READY lead ‚Üí `promote_to_ready` a QUALIFIED one clearing the gate ‚Üí else `_advance` one unit of work. `_advance` *is* the steering, and it's the qualifier's own **explore/exploit** split: **explore** (`neg ‚â§ pos`/cold start) BALD-labels the pool with **no gate** (a low-confidence lead is the most informative label; page a maximal in first if the pool is empty), **exploit** (`neg > pos`) qualifies the strongest lead clearing `min_gp_confidence` (`consumable_candidates`) or `discover`s if none clears it. The gate belongs to exploit alone ‚Äî a *spend* gate ("will this call buy an email or just park at QUALIFIED?"), the *same constant* `promote_to_ready` reads, so they can't drift. The earlier design applied it in both states and ran BALD over the confidence-filtered set ‚Äî most-uncertain lead from a bucket stripped of uncertain leads; the split removes that. Two "is this pool promising?" bars were tried earlier and both were constants in disguise ‚Äî see the `pools.py` docstring). **Keyword injection is the one mechanism** that puts queries and profiles in one embedding space: a discovered lead is embedded as `profile_text + clause_terms(its retrieving query)` (`db/leads.create_lead` ‚Üí `discovery.embed_profile`) while `profile_text` (the LLM qualifier's input) stays clean, so the GP learns query-term ‚Üí fit as a byproduct of labelling and can score a never-run query by its keywords. `EmptyClauseSet` is the anti-monotone prune: pre-screen feeds it **size-1** empties and backoff feeds **sub-maximal** ones (dropping value clauses only ‚Äî the headcount band rides every child, since loosening a bound queries off-ICP), so the subset test bites *within* one pool immediately (recording the size-1 `{location=Oman}` prunes every maximal containing Oman); a maximal recorded empty before a mint still prunes the deeper maximals a later mint grows. See the roadmap cards `p2-e3-discovery-unified-gp-query-selection` and `p2-e3-discovery-empty-set-backoff` (both supersede `p2-e3-discovery-query-graph-search`).
- **Contacts store (hub)**: `contacts/service.py` ‚Äî an optional free `profile_url ‚Üí email` cache at `hub.openoutreach.app`, tried *before* the paid finder and given back to on a fresh paid hit. Both calls are best-effort (outage/no-token ‚Üí no-op) and gated on `has_mailbox()`. The give-back is **non-EEA only** and derived from the operator's onboarding country (`not is_eea_located`) ‚Äî never a stored toggle; the server re-gates authoritatively. A per-operator token is earned on first contribution and stored in `SiteConfig.contacts_api_token` (never the repo).
- **Config**: `SiteConfig` DB singleton ‚Äî `ai_model` (a pydantic-ai `provider:model` id, e.g. `anthropic:claude-sonnet-4-5-20250929`; bare `gpt-*`/`o1`/`o3`/`claude-*`/`gemini-*` auto-prefixed), `llm_api_key`, `llm_api_base` (only for `openai_compatible:*`), `bettercontact_api_key` (blank disables discovery + enrichment), `contacts_api_token`/`contacts_api_url`, `country_code` (the only persisted operator setting ‚Äî drives timezone + email jurisdiction). `core/conf.py`: `CAMPAIGN_CONFIG` (ML + human-rhythm defaults, incl. `min_gp_confidence` for the paid-step rank gate ‚Äî which is **only** a spend gate; there is no discovery-interleave threshold), `COLLECT_BACKOFF_BASE_S`/`COLLECT_BACKOFF_MAX_S`/`COLLECT_DEADLINE_S` (the `collect_email` poll backoff ‚Äî paid spend is gated by send-headroom, not a cap), `DEFAULT_EMAIL_DAILY_LIMIT`, `BCC_OPERATOR_ON_SEND` (BCC the operator's own inbox on every send ‚Äî **off by default**), active-hours knobs (`ENABLE_ACTIVE_HOURS`/`ACTIVE_*`; `ACTIVE_TIMEZONE=None` ‚Üí resolved from the operator's country).
- **Django apps** (all nested under `openoutreach/`, dotted `AppConfig.name`, short labels): `core` (engine ‚Äî daemon, task queue + scheduler, Campaign/SiteConfig/Task, llm, onboarding, ML, discovery/qualify pipeline, the two agents), `crm` (Lead + Deal), `chat` (ChatMessage ‚Äî the per-Deal conversation), `emails` (discovery/enrichment client, Mailbox + import + SMTP/IMAP, sender, the three task handlers), `legacy` (model-less migration-history anchor). `contacts` is a service-only module (no models, not an installed app). One engine, one channel.
- **Data dir**: `data/` holds `db.sqlite3`. Docker mounts a volume at `/app/data`.
- **Docker**: `python:3.12-slim` multi-stage build with `uv`; no browser, no VNC. `compose/openoutreach/Dockerfile`. `BUILD_ENV` arg selects requirements.
- **CI/CD**: `.github/workflows/tests.yml` (pytest), `deploy.yml` (**every push to `main`**, plus `v*` tags ‚Üí `make docker-test` ‚Üí build + push `ghcr.io/eracle/openoutreach` tagged `latest` + `sha-<pushed tip>` ‚Üí `repository-dispatch: image-updated` to the hub repo). **There is no release gate: merging to `main` republishes `:latest`**, so code *and schema migrations* reach anyone pulling `latest` on merge, not on a tag. No `v*` tag has ever been cut, so no semver tag exists to pin or roll back to.
